# [FR-015] Orientation-Aware Start-State Modeling

## Labels
`Future`, `product`, `planner`, `analytics`, `case-model`

## User Story
As a cuber, I want the trainer to distinguish the same case by start orientation, slot, and angle, so that recognition, execution, and algorithm guidance reflect the real difficulty I feel in solves.

## Proposed Solution

### Overview
Extend the trainer from case-level tracking to start-state-variant tracking.

The same named case should not always be treated as one undifferentiated bucket. In practice, a PLL that starts after a required `U` or `U2` can cost measurable time, and an F2L pair in back-right can feel very different from the same pair in front-left. The trainer should model those variants explicitly when they matter.

### Implementation Flow
1. Define a portable start-state variant model for current and future case families.
2. Capture recognition and execution signals separately for each variant.
3. Update analytics, weakness scoring, and recommendations to operate at variant level when data quality is sufficient.
4. Surface algorithm guidance that can recommend a better algorithm or a better pre-turn strategy for that exact variant.

### Technical Approach
Represent a case as:
- base family or case ID
- variant metadata
- recognition metrics
- execution metrics
- algorithm recommendations and provenance

Examples:
- PLL: base case `PLL-Ja` plus variant metadata like `startingAUF`, `requiredPreTurn`, `finalAUF`, `viewAngle`
- F2L: base pair pattern plus variant metadata like `pairLocation`, `slotTarget`, `cubeRotationNeed`, `frontBackBias`

### Key Considerations
- Variant modeling must remain portable and not depend on DOM or csTimer globals in the domain layer.
- Recognition time and execution time should remain separable; otherwise the model will blur two different human bottlenecks.
- The feature should degrade gracefully when a family does not yet have reliable variant extraction logic.
- Recommendation quality depends on catalog provenance and alternate algorithm support.

## Acceptance Criteria

- [ ] A portable variant model is defined for case families with meaningful start-state differences.
- [ ] PLL orientation or AUF context can be tracked separately from the base PLL name.
- [ ] Future F2L work is explicitly expected to track pair location, slot target, and other meaningful start-state differences.
- [ ] Recognition and execution metrics are modeled as separate signals for each variant.
- [ ] Recommendation logic can suggest either a better algorithm or a better pre-turn strategy for a specific variant.
- [ ] The feature remains future-scoped and does not silently distort current case-level stats until variant capture is implemented cleanly.
