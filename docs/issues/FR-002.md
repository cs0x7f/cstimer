# [FR-002] Structured Training Plans

## Labels
`MUS`, `product`, `trainer-plans`

## User Story
As a cuber, I want reusable training templates and guided setup, so that I can practice with a plan instead of inventing sessions manually.

## Proposed Solution

### Overview
Define guided v1 templates that turn broad goals into concrete session structures.

### Implementation Flow
1. Specify supported plan archetypes for v1.
2. Define setup inputs such as session length and training focus.
3. Describe the output plan structure used by the drill queue.

### Technical Approach
Use domain-level `TrainingPlan` and `DrillDefinition` concepts that can later migrate out of csTimer.

### Key Considerations
- Keep plan setup simple enough for returning cubers.
- Avoid exposing too many knobs in v1.

## Acceptance Criteria

- [x] V1 templates are explicitly named and bounded to the supported v1 goals (returning cuber, last-layer focus, cross focus, or an approved equivalent set). **Implemented: `quick-review`, `full-rotation`, and `weakness-blitz` in `trainer-setup.js`, with cross entry reusing the same template shell through cross-specific copy and `Cross` case-set defaults.**
- [x] Setup inputs specify the exact configurable fields exposed in v1 and any allowed ranges or enums. **Implemented: session name (free text), case set (`PLL` / `OLL` / `Both` for last-layer flows, `Cross` for cross flows), session length slider (`5-78` for last-layer or returning plans, `5-20` for cross plans), adaptive weighting (toggle), show hints (toggle), repeat failed (toggle), and randomize order (toggle).**
- [x] `TrainingPlan` output shape is documented with named fields needed by the drill queue (template ID, goal, drill blocks, session length, and focus settings at minimum). **Implemented: `_buildTrainingPlan()` returns `{planId, templateId, name, goal, totalAttempts, drillBlocks: [{blockId, type, caseSet, caseIds, targetCount}], focusSettings: {weakCaseBias, coverageFloor, sessionMode, showHints, repeatFailed, randomizeOrder}, createdAt}`. Plans are saved via `trainerIntegration.savePlan()`, marked active via `trainerIntegration.setActivePlanId()`, and `Start Session` attempts `trainerIntegration.startSession()` before routing into the active trainer surface.**
- [x] The issue makes clear which plan choices are intentionally out of scope for v1 so later agents do not add extra knobs by assumption. **Out of scope for v1: custom drill block composition, per-case algorithm editing, multi-block session sequencing, cross-specific time targets, OLL subset selection, and custom weighting profiles. These are documented in `TrainerV1.md` V1 Boundaries section.**
