# [FR-008] Standalone TypeScript Extraction Path

## Labels
`Future`, `architecture`, `typescript`

## User Story
As a maintainer, I want the trainer domain documented independently of csTimer internals, so that it can later move into a modern TypeScript app.

## Proposed Solution

### Overview
Document domain contracts and architecture boundaries that allow future extraction into Next.js plus optional Convex.

### Implementation Flow
1. Define trainer domain models independent of current UI details.
2. Identify csTimer-specific adapters versus portable domain logic.
3. Define migration expectations without making extraction a v1 blocker.

### Technical Approach
Separate product contracts from implementation details and avoid baking PHP/Closure-era assumptions into trainer concepts.

### Key Considerations
- This is a future architecture track, not a v1 rewrite task.
- V1 docs should still make extraction realistic later.

## Acceptance Criteria

- [ ] Portable domain concepts are documented.
- [ ] csTimer-specific versus portable concerns are separated.
- [ ] Future extraction is scoped without being required for v1.
