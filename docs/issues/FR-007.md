# [FR-007] Source-Backed Case Catalog

## Labels
`MUS`, `content`, `catalog`, `provenance`

## User Story
As a maintainer, I want normalized case content with provenance notes, so that training content can be curated responsibly and updated over time.

## Proposed Solution

### Overview
Create a structured content model for PLL, OLL, and cross references that records origin without copying source material wholesale.

### Implementation Flow
1. Define the v1 catalog shape and scope.
2. Define provenance fields for algorithms, notes, and trust level.
3. Define the handoff format for user-fetched source packets and later normalization.

### Technical Approach
Use structured fields for case ID, source, recommendation notes, alternates, and confidence rather than storing scraped content dumps.

### Key Considerations
- Sources should remain traceable.
- Content normalization should support future updates and disagreements between sources.

## Acceptance Criteria

- [ ] Catalog scope is defined for v1.
- [ ] Provenance fields are documented.
- [ ] User source-packet format is documented.
