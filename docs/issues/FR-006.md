# [FR-006] Local-First Persistence and Export Compatibility

## Labels
`MUS`, `platform`, `storage`, `offline`

## User Story
As a cuber, I want trainer data to work offline and survive export/import, so that the training layer feels like part of csTimer instead of a fragile add-on.

## Proposed Solution

### Overview
Plan trainer persistence separately from raw solve records while ensuring export/import compatibility.

### Implementation Flow
1. Define trainer data concepts and storage ownership.
2. Define export/import expectations for trainer data.
3. Define offline-first guarantees and non-goals.

### Technical Approach
Use local-first storage, separate trainer keys, and an explicit compatibility plan with current csTimer export behavior.

### Key Considerations
- Trainer data must not corrupt existing solve storage.
- Export/import support must be planned early, not retrofitted.

## Acceptance Criteria

- [x] Trainer data storage model specifies which browser storage mechanism (localStorage, IndexedDB) and key naming convention. ✅ Implemented through `storage-adapter.js` on top of csTimer's `storage.getKey()` / `storage.setKey()` abstraction, with `trainer:`-prefixed keys and fallback-safe registration in `kernel.js`
- [x] Export/import format is defined as a concrete JSON schema that can be validated. ✅ Implemented as a `trainer` export block in `export-bridge.js` with versioned validation and explicit profile/plans/stats/sessions fields
- [x] Documented list of what happens to trainer data during csTimer's existing export/import flow. ✅ Solve-only imports preserve existing trainer data, valid trainer imports overwrite trainer sections, empty trainer blocks reset trainer data to defaults, invalid trainer blocks log warnings without breaking solve import, and trainer data now rides along with the same csTimer/WCA server backup flow and Google Drive backup flow as the rest of the export payload
- [x] Offline behavior specifies which features work without network and what data is lost if the browser clears storage. ✅ Trainer remains offline-first through browser-local persistence; clearing site data removes trainer data just like solve data, which is documented in `docs/architecture/persistence-plan.md`
