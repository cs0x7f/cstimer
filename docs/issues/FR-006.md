# [FR-006] Local-First Persistence and Export Compatibility

## Labels
`MUS`, `platform`, `storage`, `offline`

## User Story
As a cuber, I want trainer data to work offline and survive export/import, so that the training layer feels like part of csTimer instead of a fragile add-on.

## Proposed Solution

### Overview
Plan trainer persistence separately from raw solve records while ensuring export/import compatibility.

### Implementation Flow
1. Define trainer data concepts and storage ownership.
2. Define export/import expectations for trainer data.
3. Define offline-first guarantees and non-goals.

### Technical Approach
Use local-first storage, separate trainer keys, and an explicit compatibility plan with current csTimer export behavior.

### Key Considerations
- Trainer data must not corrupt existing solve storage.
- Export/import support must be planned early, not retrofitted.

## Acceptance Criteria

- [ ] Trainer persistence concepts are documented.
- [ ] Export/import expectations are documented.
- [ ] Offline behavior and compatibility risks are called out explicitly.
